# ADR-007: LLM Access Strategy

**Статус:** ACCEPTED (подтверждено в Beta-0.1b)  
**Дата:** 2026-06-17

## Контекст
Для интеграции с локальными LLM-моделями необходимо определить способ взаимодействия между оркестратором (Python/LangGraph) и моделями (llama.cpp).

## Решение
Все обращения к локальным моделям выполняются через **OpenAI-compatible API** llama-server. **Прямой вызов llama.cpp из Python запрещён в production-path Agent-Developer.**

### Обоснование:
1. **Стандартизация:** OpenAI API является де-факто стандартом для взаимодействия с LLM
2. **Гибкость:** Возможность замены бэкенда (llama.cpp → vLLM → Ollama) без изменения кода оркестратора
3. **Упрощение:** HTTP API проще для отладки и мониторинга, чем прямой вызов C++ библиотек
4. **Совместимость:** Многие библиотеки (LangChain, LlamaIndex) имеют встроенную поддержку OpenAI API
5. **Безопасность:** Изоляция низкоуровневых вызовов в отдельном процессе (llama-server)

### Технические детали:
- **Endpoint:** `http://127.0.0.1:8081/v1/chat/completions`
- **Метод:** HTTP POST
- **Формат:** JSON (OpenAI-compatible schema)
- **Библиотека:** `requests` (Python)
- **Timeout:** 120 секунд

## Критерий пересмотра
Если потребуется поддержка проприетарных API (OpenAI, Anthropic) или специфических возможностей llama.cpp, недоступных через OpenAI API, данное решение может быть пересмотрено.

## Последствия
- **Плюс:** Единый интерфейс для всех LLM-вызовов, возможность замены бэкенда
- **Минус:** Небольшой оверхед на HTTP-сериализацию (незначительно для локальных моделей)
- **Ограничение:** Зависимость от доступности llama-server
