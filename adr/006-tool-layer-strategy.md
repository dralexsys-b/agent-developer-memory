# ADR-006: Стратегия Tool Layer для Beta-0.1a

**Статус:** ACCEPTED (Beta-0.1a scope)  
**Дата:** 2026-06-16

## Контекст
Beta-0.1a представляет собой минимальный вертикальный срез: генерация одного Python-файла и его валидация через `py_compile`. Необходимо определить способ выполнения операций с файловой системой (запись, компиляция).

## Решение
1. Для Beta-0.1a отдельный модуль `tools.py` **НЕ создаётся**.
2. Операции записи файла, компиляции и валидации реализуются напрямую в файле `orchestrator_v2_beta_0_1a.py`.
3. **Важное условие:** Эти операции оформлены как отдельные, изолированные функции (в частности, `execute_write_and_compile`), а не как встроенный код в узлах LangGraph. Это изолирует побочные эффекты и позволит при переходе к Beta-1 вынести функции в `tools.py` без изменения бизнес-логики оркестратора.

## Критерий пересмотра
Если одна и та же операция начинает использоваться более чем в одном узле, или появляется необходимость отдельного тестирования операций вне контекста оркестратора, создаётся `tools.py` и данное решение пересматривается (ADR-007).

## Подтверждение
Функция `execute_write_and_compile` реализована в `src/orchestrator_v2_beta_0_1a.py` как изолированный сайд-эффект. Узел `write_and_validate_node` вызывает эту функцию по имени и преобразует состояние графа исключительно на основе её возвращаемого словаря. Цикл записи/валидации и корректность этого разделения подтверждены логами:
- `evidence/2026-06-16_1425_beta01a_run.log`
- `evidence/2026-06-16_1800_beta01a_repro_run.log`

## Последствия
- **Плюс:** Максимальная скорость доставки Beta-0.1a, отсутствие преждевременных абстракций, код остаётся читаемым.
- **Минус:** Требует дисциплины от разработчика, чтобы функции оставались чистыми и готовыми к будущему извлечению.
- **Ограничение:** Данное решение подтверждено только для Beta-0.1a vertical slice и может быть пересмотрено для Beta-0.2+ в соответствии с критерием пересмотра.
