## Purpose

PROGRAM_DESIGN_PASSPORTS является Canonical Source инженерных паспортов документов репозитория.

Каждый паспорт представляет собой аналитическую модель отдельного документа и создаётся на основании полного анализа его содержимого.

PROGRAM_DESIGN использует PROGRAM_DESIGN_REGISTRY как единственный источник информации о документах, а PROGRAM_DESIGN_REGISTRY строится исключительно на основе сведений, содержащихся в настоящем документе.

-------------------------------------------------------------------------------

## Passport Construction Standard

### Purpose

Цель паспорта — представить инженерную модель документа, позволяющую понять его роль, ответственность и связи без повторного чтения полного текста документа.

Паспорт не заменяет исходный документ и не является его Canonical Source.

---

### Construction Rules

Каждый паспорт составляется **исключительно после полного прочтения анализируемого документа**.

Запрещается составлять паспорт:

- по названию документа;
- по структуре репозитория;
- по предыдущим версиям паспорта;
- по предположениям;
- по содержимому других документов.

Каждое инженерное утверждение паспорта должно быть подтверждено содержимым анализируемого документа.

Предположения, реконструкция отсутствующей информации и свободная интерпретация запрещены.

---

### Analysis Goal

При анализе документа необходимо определить:

- для чего существует документ;
- что именно он определяет;
- какова его инженерная ответственность;
- какие темы являются основными;
- от каких документов он зависит;
- какие документы могут зависеть от него;
- какие инженерные выводы следуют из его содержимого;
- какие открытые вопросы остаются после анализа;
- какое место документ занимает в архитектуре репозитория.

Паспорт должен позволять понять инженерную роль документа без повторного чтения его полного содержимого.

---

### Mandatory Passport Sections

Каждый паспорт должен содержать следующие обязательные разделы:

- Назначение
- Основная ответственность
- Тип документа
- Основные разделы
- Входящие зависимости
- Исходящие зависимости
- Наблюдения
- Открытые вопросы

---

### Analysis Principles

При составлении паспорта необходимо:

- описывать фактическое содержимое документа;
- не повторять текст документа;
- выделять инженерную семантику;
- отделять факты от предположений;
- явно фиксировать границы ответственности документа;
- выявлять реальные зависимости, даже если они явно не перечислены;
- использовать терминологию самого документа, если она не противоречит Canonical Source;
- каждый паспорт анализируется независимо от остальных паспортов.

---

### Relationship to PROGRAM_DESIGN_REGISTRY

PROGRAM_DESIGN_REGISTRY создаётся и поддерживается инженером вручную на основании настоящих паспортов. Registry содержит только сведения, содержащиеся в настоящем документе и необходимые алгоритму PROGRAM_DESIGN.

PROGRAM_DESIGN_REGISTRY является операционной проекцией настоящих паспортов и содержит только сведения, необходимые алгоритму PROGRAM_DESIGN.

Любое изменение паспорта является основанием для актуализации PROGRAM_DESIGN_REGISTRY.

Любые сведения, отсутствующие в паспортах, не могут быть включены в Registry.

---

## Engineering Invariants

1. Каждый паспорт создаётся исключительно после полного анализа соответствующего документа.

2. Каждый паспорт является аналитическим представлением одного документа.

3. Паспорт не заменяет исходный документ и не является его Canonical Source.

4. Любое изменение документа требует актуализации соответствующего паспорта.

5. PROGRAM_DESIGN не использует паспорта напрямую.

6. PROGRAM_DESIGN_REGISTRY формируется исключительно на основе настоящих паспортов и является единственным источником данных для PROGRAM_DESIGN.

В отличие от остальных документов репозитория он не описывает архитектурные правила или инженерные процессы, а отвечает исключительно на вопрос:

> «В каком состоянии проект находится сейчас?»

Первая часть фиксирует общие сведения:

- название проекта;
- активный репозиторий;
- текущий слой;
- дату состояния.

Затем определяется текущий инженерный фокус.

На момент документа основным направлением является реализация первой итерации Runtime:

- Core Foundation;
- Event Bus;
- Runtime Engine;
- FSM;
- Context Manager.

Следующий раздел представляет состояние всей архитектуры по слоям.

Для каждого слоя фиксируется степень готовности:

- Governance;
- Information Architecture;
- Operational Architecture;
- Runtime Implementation;
- Engineering Knowledge.

Это позволяет за несколько секунд понять степень завершённости всей архитектуры проекта.

После этого документ ведёт журнал последних крупных изменений.

Он фиксирует последовательность появления основных архитектурных документов и дорожной карты Runtime.

Последний раздел определяет ближайшие инженерные задачи.

На момент документа следующими шагами являются:

- создание runtime-репозитория;
- реализация первой итерации Runtime;
- написание базовых тестов;
- переход к интеграции Memory.

Таким образом документ выполняет функцию инженерной панели состояния проекта.

Это единственный документ, который должен регулярно изменяться по мере развития проекта.

Документ вводит полноценную инженерную модель управления программами проекта.

В отличие от ENGINEERING_PLAYBOOK, который описывает инженерный процесс разработки, данный документ управляет более крупной единицей планирования — инженерной программой.

Сначала определяется место Program Management в архитектуре документации и связь с другими документами проекта.

Далее вводится собственная терминология.

Определены основные понятия:

- Program;
- Program Plan;
- Phase;
- Milestone;
- Backlog;
- Review;
- Retrospective.

Каждому термину даны:

- определение;
- инженерная роль в системе управления.

Центральной частью документа является жизненный цикл программы.

Определены последовательные состояния:

- Draft;
- Architecture Review;
- Frozen;
- In Progress;
- Review;
- Completed;
- Archived.

Дополнительно фиксируются разрешённые и запрещённые переходы между состояниями.

Особое внимание уделено правилу Frozen Plan.

После перехода программы в состояние Frozen запрещается изменять:

- область программы;
- фазы;
- контрольные точки;
- критерии завершения.

Все новые идеи должны переноситься в Backlog будущих программ.

Документ также вводит один из ключевых организационных инвариантов проекта:

Single Active Program.

Одновременно может выполняться только одна инженерная программа.

Это позволяет избежать конкурирующих архитектурных изменений и сохранить концентрацию инженерных усилий.

Следующий раздел описывает правила управления Backlog.

Backlog рассматривается как отдельное хранилище будущих инженерных инициатив и не смешивается с выполняемой программой.

После этого определяется обязательная модель оценки программ.

Каждая программа обязана пройти:

- Review перед завершением;
- Retrospective перед архивированием.

Завершается документ двумя глобальными инженерными инвариантами:

- One Program Per Goal;
- Review Informs Future Planning.

Документ задаёт единый стандарт управления инженерными программами и отделяет стратегическое планирование от непосредственного выполнения инженерной работы.

  - PHASE 3: Missing Responsibilities Detection

DOCUMENT: PROGRAM_DESIGN_EXECUTION.md
===============================================================================
Назначение
Определяет модель детерминированного исполнения Approved Frozen Program через последовательность Execution Plan.
Стандарт описывает не процесс исполнения, а модель исполнения: какие артефакты существуют, какими свойствами они обладают, как связаны между собой и при каких условиях происходит переход от одного к другому.
Стандарт независим от исполнителя (человек, LLM, агент, CI/CD система) и среды исполнения (чат, IDE, терминал).

---
Основная ответственность
Определяет:
- Execution Model с двумя сущностями (Approved Frozen Program, Execution Plan);
- Relations между сущностями (references, ordered sequence, identifies bounded scope);
- Core Concepts (Approved Frozen Program, Execution Plan, Execution Context, Execution Scope, Completion Conditions);
- Invariants (Program Immutability, Plan Immutability, Ordered Execution, Scope Isolation, Context Continuity);
- Execution Plan Specification (Required Sections, Invariants);
- Lifecycle (States, Transitions, Immutability Rule);
- Execution Plan Transition (Transition Rule, Context Inheritance, Continuity Invariant);
- Examples (Context Continuity, Single Execution Plan, Scope Isolation);
- Conformance criteria.

Стандарт разделён на две части:
- PART I: CORE — стабильная аксиоматическая модель (сущности, отношения, инварианты);

- предсказуемость выполнения графа;

- упрощение отладки;

Наблюдения

Документ определяет общий инженерный стандарт, распространяющийся на всю экосистему Agent Developer.

Первая часть формулирует пять фундаментальных инженерных правил.

Rule -1 утверждает, что архитектура должна служить реализации, а не наоборот. Если архитектурное решение усложняет код без практической пользы, архитектура подлежит пересмотру.

Rule 0 закрепляет принцип предпочтения простой архитектуры сложной реализации. Документ рекомендует создавать новые классы, пакеты и явные зависимости вместо накопления условной логики и скрытого поведения.

Rule 1 запрещает преждевременное создание структуры проекта. Каталоги появляются только одновременно с первой рабочей реализацией.

Rule 2 требует, чтобы любой shell-скрипт, используемый более одного раза, сохранялся в репозитории как отдельный файл, а не оставался в истории переписки.

Rule 3 повторяет принцип изоляции Bootstrap. Зависимость допускается только в направлении Bootstrap → Runtime во время создания репозитория.

Следующий раздел посвящён требованиям к качеству кода.

Документ устанавливает строгие ограничения, включая:

- обязательную строгую типизацию;
- запрет необоснованного использования Any, Dict и Optional;
- обязательное прохождение `mypy --strict`;
- использование ruff;
- обязательное покрытие публичных компонентов тестами;
- отсутствие глобального состояния;
- использование абсолютных импортов;
- явное объявление `__all__`;
- обязательные docstrings.

Последний раздел определяет правила инженерного процесса.

Документ стандартизирует последовательность разработки:

Roadmap → Implementation → Tests → Review → Lessons → Architecture Update.

Особое внимание уделяется архитектурным ограничениям.

В частности, документ требует:

- Contracts First;
- отсутствия God Objects;
- строгого разделения ответственности Event Bus;
- минимальной ответственности Registry;
- однонаправленных зависимостей между слоями;
- обязательного Review перед изменением архитектуры.

В отличие от шаблонов Bootstrap данный документ содержит полноценные инженерные нормы и является нормативным руководством для разработки всех компонентов проекта.

В начале файла явно сформулированы ограничения его применения:

- Git не инициализируется автоматически.
- Пользователь самостоятельно выбирает систему контроля версий.
- Соблюдается правило Bootstrap Isolation (Rule 3): Bootstrap никогда не импортирует Runtime.

Сценарий выполнения построен как последовательность независимых этапов:

1. проверка аргументов командной строки;
2. проверка наличия каталога шаблонов;
3. проверка существования и пустоты целевого каталога;
4. создание стандартной структуры каталогов;
5. копирование базовых шаблонов.

Для обработки ошибок используется единая функция `die()`, завершающая выполнение с диагностическим сообщением.

Скрипт использует строгий режим Bash (`set -euo pipefail`), что предотвращает продолжение выполнения при ошибках.

После успешного завершения выводится рекомендуемая последовательность следующих действий:

- переход в новый каталог;
- при необходимости инициализация Git;
- настройка README;
- создание структуры исходного кода;
- первый commit.

Документ реализует принцип стандартизированного создания новых репозиториев без привязки к конкретной реализации Runtime.

Ключевые наблюдения

Данный лог является первым практическим подтверждением полностью автоматизированного self-healing цикла Agent Developer.

Во время первой генерации оркестратор намеренно возвращает синтаксически некорректный код (`FORCE_INVALID_CODE_ON_FIRST_ATTEMPT=True`), после чего валидатор обнаруживает ошибку компиляции и передаёт её обратно в LLM вместе с предыдущим кодом.

На второй итерации модель успешно исправляет программу, валидатор подтверждает корректность результата, после чего задача завершается со статусом `completed`.

Лог демонстрирует согласованную работу всех основных компонентов Beta-0.2.r1: генератора кода, валидатора, маршрутизатора, механизма повторных попыток и финального агрегатора результата.

Кроме того, документ фиксирует производительность модели, структуру Final State, параметры retry-механизма и корректную очистку временных файлов после успешного завершения выполнения.
